## Короткий вердикт

Prompt/context слои в целом соответствуют research для Demo MVP: они короткие, не тащат полный research в context window и удерживают ассистента в режиме проекта ТК/ТТК, предупреждений, статусов данных и нейтрального JSON.

Главная проблема не в многословии, а в недоснятой конкретике. Research про справочники, версии данных, consumer/menu-информацию и критерии выбора ТК/ТТК перенесен слишком мягко. Из-за этого LLM может формально следовать промту, но давать менее проверяемый и менее полезный результат.

## Лаконичность

Слои лаконичные. Каждый markdown layer занимает 7-23 строки и примерно 94-154 слова. Это хороший размер для статического контекста.

Дублирование есть, но умеренное:

- проектный статус документа повторяется в `00_AGENT_ROLE.md`, `01_PRODUCT_RULES.md`, `06_OUTPUT_RULES.md`, `07_DEMO_LIMITATIONS.md`;
- запрет на прямой импорт повторяется в `00_AGENT_ROLE.md`, `06_OUTPUT_RULES.md`, `07_DEMO_LIMITATIONS.md`;
- предупреждения и статусы повторяются в нескольких слоях.

Для Demo MVP это допустимо: повторяются safety-critical границы. Сокращать стоит только после проверки реальных ответов LLM.

## Полезность

Полезность высокая по ядру продукта:

- ассистент формирует проект, а не финальный юридический документ;
- различает ТК и ТТК;
- не должен выдумывать сроки хранения, температуры, безопасность и БЖУ;
- показывает warnings и data statuses;
- не обещает прямой импорт в iiko, r_keeper, StoreHouse или 1С;
- держит production state machine, semantic context и calculation engine вне Demo MVP.

Полезность проседает там, где prompt остается на уровне "показывай предупреждения" без проверяемого правила, когда именно и с каким статусом предупреждать.

## Findings

### 1. Слабое правило по справочникам и БЖУ

Research требует, чтобы расчет БЖУ и справочные значения имели проверяемый источник, версию и дату актуализации. В prompt layers это сведено к статусу "Взято из справочника".

Риск: ассистент может рассчитать или назвать БЖУ из общих знаний без воспроизводимого источника.

Рекомендуемая правка:

- в `05_DATA_STATUSES.md` добавить: если значение взято из справочника, указывай источник, версию или дату актуализации, если они известны;
- в `06_OUTPUT_RULES.md` добавить: БЖУ и калорийность не считать уверенно без справочника; без справочника ставить `требует проверки` или `не заполнено`.

### 2. Недоперенесен consumer/menu вывод

Research говорит, что документ должен помогать не только готовить блюдо, но и поддерживать корректное описание для меню, доставки и внутренних процессов.

В prompt layers есть доставка и упаковка, но нет явного правила про consumer-facing сведения: состав, масса порции, способ приготовления, пищевая ценность, предупреждения о неполных данных.

Риск: итоговый проект может стать внутренней технологической заготовкой, но хуже помогать меню/доставке.

Рекомендуемая правка:

- в `04_TECH_CARD_STRUCTURE.md` добавить блок "сведения для меню/доставки, если применимо";
- в `03_NORMATIVE_BASE_SHORT.md` добавить одну строку про понятную информацию для потребителя без юридического перегруза.

### 3. Критерий выбора ТК/ТТК слишком мягкий

Сейчас prompt говорит "чаще предлагай ТТК" и "может быть достаточно ТК". Research дает более четкую развилку: стандартная/готовая рецептура - ТК; новая, авторская, адаптированная, измененная под формат заведения или национальный акцент - ТТК.

Риск: модель будет колебаться или выбирать тип документа по стилю ответа, а не по предметному критерию.

Рекомендуемая правка:

- в `01_PRODUCT_RULES.md` заменить мягкие формулировки на явное правило выбора;
- в `02_WORKFLOW.md` на шаге выбора типа документа требовать короткое обоснование выбора.

### 4. Canonical JSON обозначен, но не закреплен как форма полезного результата

Research предлагает canonical JSON как нейтральное представление блюда, рецептуры, расчета, пищевой ценности и предупреждений. Prompt layers говорят "нейтральный и минимальный", но не перечисляют ожидаемые блоки.

Runtime fallback `structured_json_from_draft` сейчас возвращает только `title`, `document_type`, `project_status` и `sections`. Это не противоречит Demo MVP, но слабее research-вывода.

Риск: JSON будет формальным контейнером, а не полезным промежуточным форматом для будущих интеграций.

Рекомендуемая правка:

- в `06_OUTPUT_RULES.md` добавить минимальный список expected JSON blocks: dish, ingredients, process, nutrition, warnings, data_statuses, version;
- не превращать это в схему конкретной учетной системы.

### 5. Есть общие слова, но они не доминируют

Примеры общих формулировок:

- "Отвечай пользователю кратко, понятно и практично";
- "Не перегружай пользователя юридическими деталями";
- "Показывай предупреждения".

Они не критичны, потому что рядом есть конкретные предметные правила. Но лучше заменить часть общих слов на операционные критерии:

- не цитируй нормы, если пользователь не просит;
- показывай практический вывод и статус данных;
- предупреждай явно по срокам, температурам, БЖУ, потерям, безопасности и неполным данным.

## Что не стоит делать

- Не вставлять полный research в prompt layers.
- Не добавлять длинные юридические объяснения в `03_NORMATIVE_BASE_SHORT.md`.
- Не делать production JSON schema в markdown prompt layer.
- Не добавлять интеграционные mapping под iiko, r_keeper, StoreHouse или 1С.
- Не убирать повтор safety-critical ограничений до проверки реальных ответов.

## Приоритет правок

1. Усилить `05_DATA_STATUSES.md` правилом source/version/date для справочников.
2. Усилить `06_OUTPUT_RULES.md` правилом "БЖУ только при проверяемом источнике, иначе статус".
3. Уточнить правило выбора ТК/ТТК в `01_PRODUCT_RULES.md`.
4. Добавить consumer/menu block в `04_TECH_CARD_STRUCTURE.md`.
5. Добавить минимальные expected blocks для neutral/canonical JSON в `06_OUTPUT_RULES.md`.

## Итоговая оценка

Соответствие research: 7/10.

Лаконичность: 8/10.

Полезность: 7/10 сейчас, 8/10 после точечных правок выше.

Основной вывод: слои уже здоровые по форме и границам MVP. Их не нужно раздувать. Нужно добавить 5-8 конкретных правил из research, чтобы ответы были менее общими и лучше проверялись.
