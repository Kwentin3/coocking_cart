## 1. MVP-Safe One-Liner

«ТехКухня» - AI-ассистент, который помогает подготовить проект ТК/ТТК с расчетной основой, предупреждениями и статусами данных.

Использовать, когда copy должен говорить только о текущем MVP/current capability.

## 2. Target Product One-Liner

«ТехКухня» - управляемая система для технологических карт, рецептур, расчетов, документов и стандартизации процессов общепита.

Использовать, когда лендинг говорит о целевом продукте. Требование: не создавать впечатление, что все target/roadmap capabilities уже доступны сейчас.

## 3. Recommended Public Positioning

Базовая формула для лендинга:

> ТехКухня помогает кухне, технологу и управляющему превращать рецептуры в управляемые технологические документы, расчеты и проверяемые данные без хаоса в таблицах.

Эта формула шире MVP, но соответствует Product Vision. В подробном copy нужно различать:

* что уже есть в MVP/current implementation;
* что является target product capability;
* что находится в roadmap;
* что требует решения владельца продукта;
* что не имеет основания и является `unsupported_claim`;
* что запрещено обещать.

## 4. MVP-Safe Description

Пользователь описывает блюдо в чате, ассистент уточняет формат заведения, состав, выход, порции и технологию, помогает выбрать ТК или ТТК и формирует проект документа. Результат показывает, какие данные подтверждены, рассчитаны, взяты из справочника или требуют проверки.

MVP-safe copy всегда использует слова:

* проект;
* черновик;
* требует проверки;
* предупреждения;
* статусы данных;
* structured JSON для будущих интеграций.

## 5. Target Product Description

Целевой продукт помогает предприятиям общепита управлять рецептурами, технологическими картами, ТТК, расчетами, связанными документами и стандартами работы кухни. Система должна постепенно закрыть путь от идеи блюда и рецептуры до управляемой базы документов, расчетной основы, себестоимости, экспортов и future-интеграций.

Target product copy может использовать темы шире MVP:

* себестоимость;
* калькуляционные карты;
* производственные документы;
* стандартизация кухни;
* экспорт документов;
* интеграции;
* база знаний;
* тарифы/onboarding.

Но эти темы должны иметь maturity label in roadmap/traceability.

## 6. Primary Audience

MVP and target shared primary:

* небольшие кафе и рестораны;
* шеф-повара и су-шефы;
* технологи общепита;
* управляющие заведениями;
* владельцы малого и среднего общепита;
* dark kitchen / доставка;
* небольшие сети.

Target expansion:

* фабрики-кухни;
* цеха при общепите;
* технологические консультанты;
* интеграторы после появления integration layer.

Decision needed:

* крупные пищевые производства;
* склад/закупки as primary buyer;
* бухгалтерия as primary owner of cost accounting.

## 7. Core Jobs-To-Be-Done

MVP-safe jobs:

1. Подготовить проект ТК/ТТК из идеи блюда.
2. Собрать рецептуру, выход, порции, брутто/нетто and базовую технологию.
3. Понять, какие данные требуют проверки.
4. Передать проект ответственному лицу.

Target product jobs:

1. Управлять базой рецептур and технологических документов.
2. Стандартизировать процессы между кухней, технологом and управляющим.
3. Связать рецептуру, расчеты, документы and future учетные системы.
4. Постепенно добавить себестоимость, exports, document lifecycle and integrations.

## 8. MVP-Safe Claims

Можно использовать как current/MVP:

* «помогает подготовить проект ТК/ТТК»;
* «ведет от идеи блюда к структурированному проекту документа»;
* «уточняет состав, выход, порции and технологию»;
* «показывает предупреждения and статусы данных»;
* «помогает выбрать ТК или ТТК»;
* «формирует structured JSON для будущих интеграций»;
* «не заменяет технолога, а ускоряет подготовку черновика».

## 9. Target Product Claims

Можно использовать как product vision / target framing:

* «управляемая система для технологических карт and рецептур»;
* «единое место для документов and расчетов общепита»;
* «стандартизация рецептур and производственных данных»;
* «расчеты, документы and контроль версий вокруг блюда»;
* «себестоимость как целевой расчетный контур»;
* «калькуляционные карты как roadmap/target capability»;
* «экспорт документов как roadmap/target capability».

Ограничение: не писать так, будто all target capabilities уже реализованы.

## 10. Roadmap Claims

Можно использовать только с future/roadmap framing:

* полноценная себестоимость;
* закупочные цены;
* поставщики;
* калькуляционные карты;
* DOCX/PDF/Excel export;
* approval workflow;
* 1С/iiko/r_keeper/StoreHouse integrations;
* база знаний;
* тарифы/onboarding;
* режим технолога beyond MVP.

## 11. Vision Claims

Использовать осторожно, обычно не в core landing:

* план-меню;
* этикетки;
* накопительные ведомости;
* складской учет;
* полноценный производственный контур;
* enterprise compliance workflow;
* BI/analytics for menu and production decisions.

## 12. Forbidden Claims

Запрещено:

* «юридически утверждает документы»;
* «автоматически соблюдает СанПиН/ХАССП/ГОСТ/ТР ТС»;
* «полное соответствие требованиям без проверки»;
* «без ошибок»;
* «AI сам утверждает карту»;
* «лабораторно подтверждает показатели»;
* «готовый импорт в 1С/iiko/r_keeper/StoreHouse» без реализованного адаптера;
* «скачайте DOCX/PDF» как current CTA до реализации export;
* fake testimonials as real.

## 13. Preferred Terminology

| Термин | Каноническое использование |
| --- | --- |
| Технологическая карта / ТК | Документ или проект документа для стандартного блюда. |
| Технико-технологическая карта / ТТК | Более полный документ для новой, авторской or адаптированной продукции. |
| Проект документа | Обязательная формулировка для MVP output. |
| Рецептура | Состав and нормы закладки сырья/ингредиентов для заданного выхода. |
| Расчетная основа | Safe term для MVP вместо полноценной себестоимости. |
| Себестоимость | Target/roadmap contour, не MVP current feature. |
| Structured JSON | Нейтральное представление для будущих интеграций, не формат учетной системы. |
| AI-помощник | Интерфейс and helper, не автономный утверждающий субъект. |
| С учетом требований | Safe compliance wording. |

## 14. Banned Or Risky Terminology

| Термин / формулировка | Статус | Замена |
| --- | --- | --- |
| «готовая юридическая карта» | `forbidden_claim` | «проект карты для проверки». |
| «автоматическое соответствие» | `forbidden_claim` | «помогает учитывать требования». |
| «точная себестоимость» | `roadmap_claim` / unsafe as current | «себестоимость как целевой расчетный контур». |
| «полный учет затрат» | `decision_needed` | «расчетная основа and future cost layer». |
| «готовый импорт в 1С» | `roadmap_claim` or forbidden as current | «future integrations». |
| «скачать DOCX/PDF» | `roadmap_claim` | «посмотреть пример проекта» until export exists. |
| «AI все сделает» | `forbidden_claim` | «AI помогает собрать and структурировать данные». |
| Claim без основания в Product Vision / Roadmap | `unsupported_claim` | убрать или вынести владельцу продукта. |

## 15. How To Use This Summary

For landing:

* hero may use target product one-liner if details do not imply current availability;
* product demo cards should label current vs roadmap if needed;
* legal/compliance copy must stay cautious.

For MVP:

* use MVP-safe one-liner;
* keep project/status/warning boundary visible.

For agent validation:

* each strong claim should map to claim maturity;
* `forbidden_claim` blocks the change;
* `decision_needed` requires owner approval.
* `unsupported_claim` must get a Product Vision / Roadmap / owner-decision source before publication.
