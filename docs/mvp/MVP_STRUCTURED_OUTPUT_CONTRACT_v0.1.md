# MVP structured output contract v0.1

- Статус: минимальный контракт structured output Demo MVP
- Дата: 2026-05-24
- Контур: Demo MVP

## Назначение

Structured output нужен уже в Demo MVP, чтобы ответ LLM был управляемым: frontend может показать пользователю человекочитаемый ответ, а runtime может сохранить служебные поля для следующих тактов.

Этот контракт намеренно минимальный. Он не является production-доменной схемой, интеграционным форматом iiko/r_keeper/1С/StoreHouse или полноценной моделью документа.

Для Gemini structured output реализуется через generation config: JSON MIME type и JSON Schema/response schema. Формат ответа не должен обеспечиваться prompt-only форматированием. Prompt/context описывает задачу и доменные правила, а форма ответа задается provider adapter через schema-first structured output.

Официальная спецификация Gemini Structured Outputs: https://ai.google.dev/gemini-api/docs/structured-output

Runtime все равно должен валидировать результат: schema гарантирует синтаксически структурированный ответ, но не подтверждает юридическую, технологическую или семантическую корректность данных.

## Термины

**Structured output** - полный ответ LLM для runtime. Он включает `user_answer` и служебные поля: workflow status, known facts, open questions, warnings, data statuses, document draft, structured JSON и next step.

**user_answer** - пользовательская часть structured output. Это текст, который можно показать в основном чате.

**document_draft** - черновик ТК/ТТК внутри structured output. Он появляется только когда пользователь запросил формирование карты и данных достаточно для проекта.

**structured_json** - один из служебных блоков structured output. Он появляется только вместе с проектом ТК/ТТК и является нейтральным машинно-читаемым представлением будущего документа.

Structured JSON не является:

- форматом iiko;
- форматом r_keeper;
- форматом 1С;
- форматом StoreHouse;
- production-интеграционной схемой;
- юридически утвержденным документом.

## Минимальные поля

### user_answer

Текст, который видит пользователь.

Должен быть понятным, практичным и не перегруженным юридическими деталями. Если данных не хватает, ассистент задает вопросы. Если пользователь запросил карту после подтверждения состава и технологии, ассистент формирует проект.

### workflow_status

Текущий этап диалога.

Примеры:

- `idea_received`;
- `format_clarification`;
- `culinary_hypothesis`;
- `recipe_confirmation`;
- `technology_confirmation`;
- `document_type_selection`;
- `draft_generation`;
- `review_required`.

### known_facts

Что уже известно или подтверждено.

Примеры:

- название блюда;
- формат заведения;
- выход порции;
- количество порций;
- состав;
- подтвержденные ограничения;
- выбранный тип документа.

### open_questions

Что нужно уточнить перед следующим шагом.

Примеры:

- выход порции;
- количество порций;
- формат заведения;
- наличие доставки;
- технология приготовления;
- источник пищевой ценности;
- сроки хранения.

### warnings

Предупреждения о неполных, неподтвержденных или рискованных данных.

Примеры:

- сроки хранения не подтверждены предприятием;
- нормы потерь требуют производственной проработки;
- БЖУ рассчитаны только при наличии справочника;
- проект не является юридически утвержденной картой.

### data_statuses

Статусы важных данных.

Допустимые статусы:

- `confirmed_by_user`;
- `calculated_by_system`;
- `from_reference`;
- `needs_review`;
- `needs_production_test`;
- `missing`.

### document_draft

Черновик ТК/ТТК, если пользователь запросил формирование карты и есть минимально достаточные данные.

Если карта еще не формируется, поле может быть пустым или содержать `null`.

Черновик всегда должен быть помечен как проект, требующий проверки ответственным лицом предприятия.

### structured_json

Один из блоков structured output: структурированное представление будущего документа, если карта сформирована.

Для MVP это нейтральный JSON для будущих интеграций, а не формат конкретной учетной системы и не production-интеграционная схема.

Минимальные блоки:

- тип документа;
- блюдо;
- ингредиенты;
- технология;
- хранение и реализация;
- пищевая ценность;
- предупреждения;
- статусы данных;
- допущения.

### next_step

Что логично сделать дальше.

Примеры:

- задать уточняющий вопрос;
- попросить подтвердить состав;
- попросить подтвердить технологию;
- предложить тип документа;
- сформировать проект;
- передать проект на проверку ответственному лицу.

## Простая форма ответа

Для MVP adapter передает Gemini response schema, эквивалентную этой простой форме:

```json
{
  "user_answer": "string",
  "workflow_status": "string",
  "known_facts": {},
  "open_questions": [],
  "warnings": [],
  "data_statuses": [],
  "document_draft": null,
  "structured_json": null,
  "next_step": "string"
}
```

## Правила использования

- `user_answer` - единственное поле, которое напрямую показывается пользователю без служебного UI.
- Остальные поля являются служебной частью, которую runtime может сохранить и использовать на следующих тактах.
- Для Gemini REST API форма ответа задается через `generationConfig.responseMimeType = "application/json"` и `generationConfig.responseJsonSchema`, а не через prompt text.
- Prompt может объяснять задачу и бизнес-правила, но не должен быть механизмом управления форматом structured output.
- Если данных недостаточно, structured output должен содержать вопросы, а не выдуманный документ.
- Если данные неподтверждены, они должны получить явный статус.
- Если документ сформирован, он должен быть маркирован как проект.
- `structured_json` не должен появляться как самостоятельная замена `document_draft`.
- Если проект карты еще не сформирован, `document_draft` и `structured_json` должны оставаться пустыми или `null`.

## Не-goals

- Не делать production JSON schema.
- Не описывать интеграционные адаптеры.
- Не фиксировать SQL-структуры.
- Не заменять будущий calculation engine.
- Не обещать юридическую валидность документа.
- Не использовать structured JSON как готовый импорт в учетные системы.
