# MVP scope v0.1

- Статус: scope Demo MVP
- Дата: 2026-05-24
- Контур: Demo MVP

## Цель MVP

Demo MVP должен быстро показать продуктовую ценность AI-ассистента технологических карт: за короткий чат пользователь получает управляемый проект ТК/ТТК, предупреждения, статусы данных и structured JSON для будущих интеграций.

Это демонстрационный прототип, а не production-ready система.

## Основной пользовательский сценарий

1. Пользователь открывает чат.
2. Пользователь пишет: "хочу технологическую карту для курицы по-вьетнамски" или "хочу технологическую карту яичницы".
3. Ассистент уточняет формат заведения, выход порции, количество порций, состав, ограничения и степень адаптации блюда.
4. Ассистент предлагает кулинарную гипотезу, не выдавая ее за утвержденную рецептуру.
5. Пользователь корректирует и подтверждает состав.
6. Пользователь подтверждает общий технологический процесс.
7. Ассистент помогает выбрать ТК или ТТК.
8. После запроса пользователя ассистент формирует проект документа.
9. Ответ содержит человекочитаемый проект, предупреждения, статусы данных и structured JSON.

## Что входит в MVP

- Чатовый сценарий подготовки проекта ТК/ТТК.
- Два первых демо-сценария: адаптированное блюдо и простое блюдо.
- Уточняющие вопросы перед формированием документа.
- Разделение кулинарной гипотезы и проекта документа.
- Видимые предупреждения по неподтвержденным данным.
- Статусы данных: подтверждено пользователем, рассчитано системой, взято из справочника, требует проверки, требует производственной проработки, не заполнено.
- Простая file-driven context architecture.
- Markdown context layers для статического контекста.
- SQLite как хранилище короткой истории диалога и служебного результата тактов.
- Минимальный structured output.

## Что не входит в MVP

- Юридически финальная технологическая карта.
- Лабораторные данные и подтвержденные показатели безопасности.
- Автоматически подтвержденные сроки хранения, температурные режимы и нормы потерь.
- Прямой импорт в iiko, r_keeper, StoreHouse или 1С.
- Production state machine.
- Полноценный semantic context engine.
- Полноценный calculation engine.
- Production audit/event log.
- Сложные роли, права и маршруты согласования.
- Production storage design.

## Первые демо-сценарии

### Основной сценарий: курица по-вьетнамски

Цель: показать ценность ассистента на авторском или адаптированном блюде.

Ожидаемое поведение:

- ассистент уточняет формат заведения, доставку, выход порции и количество порций;
- предлагает кулинарную гипотезу;
- помогает заменить редкие ингредиенты на более доступные;
- маркирует блюдо как вероятный проект ТТК;
- выносит потери, сроки хранения и температурные режимы в предупреждения;
- формирует проект ТТК после подтверждения состава и технологии.

### Контрольный простой сценарий: яичница или омлет

Цель: проверить, что ассистент работает не только на красивом авторском сценарии.

Ожидаемое поведение:

- ассистент задает меньше уточняющих вопросов;
- не усложняет простое блюдо без необходимости;
- предлагает проект ТК, если блюдо стандартное и без существенной адаптации;
- все равно показывает статусы данных и предупреждения по неподтвержденным полям.

## Сознательные ограничения

- Короткая история диалога может отправляться в LLM целиком, пока диалог укладывается в 5-10 сообщений.
- Structured output нужен для управляемости, но не является production-доменной схемой.
- Расчеты в демо являются черновыми и должны явно иметь статус.
- Markdown context layers редактируются вручную как prompt/context assets.
- LLM не должна сама создавать или подключать новые context layers.
- Все production-темы остаются future track до отдельного архитектурного прохода.
