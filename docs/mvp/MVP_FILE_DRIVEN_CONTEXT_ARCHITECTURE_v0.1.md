# MVP file-driven context architecture v0.1

- Статус: архитектурный компромисс Demo MVP
- Дата: 2026-05-24
- Контур: Demo MVP

## Идея

Demo MVP использует file-driven context architecture: статические правила, роль ассистента, workflow, доменная семантика, статусы данных и ограничения демо хранятся в markdown-файлах.

В коде будущего прототипа не должно быть захардкоженных prompt texts, нормативных правил или доменной семантики. Код должен только читать согласованные markdown layers, собирать контекстное окно, добавлять динамическую историю диалога и вызывать LLM.

## Почему это подходит для Demo MVP

- Контекст можно быстро менять без изменения кода.
- Продуктовые правила остаются видимыми и ревьюируемыми.
- MVP не блокируется полноценной production-архитектурой.
- Markdown layers легко сверять с PRD, workflow, research и glossary.
- Команда может показать демо и собрать обратную связь до сложного backend-дизайна.

## Источники статического контекста

Статический context pack собирается из `docs/mvp/context/` в порядке, заданном `context_manifest.yml`.

Порядок важен: сначала роль, затем продуктовые правила, workflow, доменный контекст, структура документа, статусы, правила ответа и ограничения демо.

## Runtime-cycle

1. Пользователь отправляет сообщение.
2. Runtime загружает markdown context pack.
3. Runtime достает короткую историю диалога из SQLite.
4. Runtime добавляет последнее сообщение пользователя.
5. Runtime добавляет инструкцию вернуть полный structured output.
6. Runtime отправляет собранное окно в LLM.
7. LLM возвращает structured output: `user_answer` и служебные поля.
8. Frontend показывает пользователю `user_answer`.
9. Если сформирован проект карты, frontend или debug view показывает `warnings`, `data_statuses`, `document_draft` и `structured_json`.
10. Runtime сохраняет user/assistant сообщения и служебный structured result.

## Роль runtime

Runtime в MVP отвечает за простую оркестрацию:

- прочитать markdown-файлы;
- применить порядок из manifest;
- добавить историю диалога;
- вызвать LLM;
- сохранить результат.

Runtime не должен интерпретировать production state machine, выполнять deep validation, управлять approval workflow или запускать интеграционные адаптеры.

## Что можно менять без кода

- Формулировку роли ассистента.
- Продуктовые правила.
- Сжатый workflow.
- Короткий нормативный контекст.
- Структуру выдачи ТК/ТТК.
- Список статусов данных.
- Правила ответа.
- Demo limitations.

## Что не решает этот подход

- Он не заменяет production semantic context layer.
- Он не решает версионирование документов production-уровня.
- Он не делает расчетный движок.
- Он не подтверждает нормативные значения.
- Он не является audit/event log.
- Он не превращает `structured_json` в production-интеграционную схему.

## Граница

File-driven context architecture - осознанный MVP-компромисс. Он нужен для демонстрации идеи и проверки продуктовой ценности. После подтверждения ценности production architecture может заменить или расширить этот подход управляемым состоянием, контрактами, валидацией и аудитом.
