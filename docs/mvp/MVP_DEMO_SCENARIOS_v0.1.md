# MVP demo scenarios v0.1

- Статус: первые сценарии проверки Demo MVP
- Дата: 2026-05-24
- Контур: Demo MVP

## Назначение

Документ фиксирует первые сценарии, по которым будет проверяться Demo MVP.

Сценарии нужны не для production test suite, а для демонстрационной проверки поведения: ассистент должен вести диалог, не обещать подтвержденные данные, различать ТК и ТТК, показывать warnings/statuses и возвращать structured output.

## Сценарий 1. Курица по-вьетнамски

### Что проверяет

- Авторское или адаптированное блюдо.
- Выбор проекта ТТК.
- Уточнение формата dark kitchen / доставка.
- Учет доступности ингредиентов.
- Предупреждения по потерям, хранению и температурным режимам.
- Формирование проекта ТТК.
- Наличие structured JSON как блока structured output.

### Стартовый запрос пользователя

```text
Хочу технологическую карту для курицы по-вьетнамски.
```

### Ожидаемые уточняющие вопросы

Ассистент должен уточнить:

- формат заведения: кафе, ресторан, доставка, dark kitchen или другое;
- нужен ли сценарий доставки и упаковки;
- выход порции;
- количество порций для расчета;
- блюдо стандартное или авторская/адаптированная версия;
- желаемый уровень остроты и национального акцента;
- ограничения по редким, дорогим или труднодоступным ингредиентам;
- нужно ли заменить или сделать опциональными спорные ингредиенты.

### Ожидаемые признаки корректного поведения

- Ассистент не формирует проект карты сразу после первого сообщения.
- Ассистент предлагает кулинарную гипотезу, а не утвержденную рецептуру.
- Ассистент понимает, что адаптация под доставку и локальные ингредиенты чаще ведет к проекту ТТК.
- Ассистент выносит лемонграсс или другие редкие ингредиенты в обсуждение доступности.
- Ассистент показывает, что потери, сроки хранения, температурные режимы и условия доставки требуют проверки.
- Ассистент не обещает юридически финальную карту.

### Ожидаемый итоговый результат

После подтверждения состава и общей технологии ассистент должен сформировать:

- `user_answer` с кратким пояснением результата;
- `workflow_status`, близкий к `draft_generation` или `review_required`;
- `known_facts` по блюду, формату, порции, количеству порций и составу;
- `warnings` по неподтвержденным данным;
- `data_statuses` для выхода, потерь, хранения, БЖУ и условий доставки;
- `document_draft` как проект ТТК;
- `structured_json` как нейтральное машинно-читаемое представление проекта;
- `next_step`, например проверка технологом или производственная проработка.

### Что считается ошибкой поведения

- Ассистент сразу выдает финальную карту без уточнений.
- Ассистент выбирает ТК без объяснения, несмотря на авторскую адаптацию и доставку.
- Ассистент уверенно придумывает сроки хранения, потери или температурные режимы.
- Ассистент не показывает warnings и data statuses.
- Ассистент выдает structured JSON как готовый импорт в учетную систему.
- Ассистент обещает юридически утвержденный документ.

## Сценарий 2. Яичница или омлет

### Что проверяет

- Простое стандартное блюдо.
- Меньшее количество уточняющих вопросов.
- Возможность предложить проект ТК.
- Отсутствие чрезмерного усложнения.
- Сохранение warnings и data statuses даже для простого блюда.

### Стартовый запрос пользователя

```text
Хочу технологическую карту яичницы.
```

Допустимая вариация:

```text
Хочу технологическую карту омлета.
```

### Ожидаемые уточняющие вопросы

Ассистент должен уточнить только минимально полезные данные:

- формат заведения;
- выход порции;
- количество порций;
- состав: яйца, масло, соль, молоко или добавки;
- способ приготовления;
- нужна ли доставка или только подача на месте.

### Ожидаемые признаки корректного поведения

- Ассистент не перегружает пользователя вопросами про сложную авторскую адаптацию.
- Ассистент может предложить проект ТК, если блюдо стандартное и без существенной адаптации.
- Ассистент все равно показывает неполные данные: срок хранения, температурный режим, источник БЖУ, потери.
- Ассистент не делает вид, что простое блюдо автоматически не требует проверки.
- Ассистент сохраняет минимальный structured output.

### Ожидаемый итоговый результат

После подтверждения состава и технологии ассистент должен сформировать:

- `user_answer` с проектным результатом;
- `workflow_status`, близкий к `draft_generation` или `review_required`;
- `known_facts` по блюду, порции, количеству порций и технологии;
- `warnings` по неподтвержденным данным;
- `data_statuses` для ключевых полей;
- `document_draft` как проект ТК, если нет авторской адаптации;
- `structured_json` как нейтральное представление проекта;
- `next_step`, например проверка ответственным лицом.

### Что считается ошибкой поведения

- Ассистент чрезмерно усложняет простой сценарий и требует несоразмерно много данных.
- Ассистент без причины выбирает ТТК для стандартной яичницы.
- Ассистент не показывает warnings и statuses, потому что блюдо простое.
- Ассистент выдает документ как финально утвержденный.
- Ассистент не возвращает structured output.

## Общие правила проверки

Для обоих сценариев корректный MVP должен:

- не хардкодить доменные prompts в коде;
- читать markdown context layers через manifest;
- хранить историю user/assistant в SQLite;
- собирать context window на каждый такт;
- возвращать structured output;
- показывать `user_answer`;
- сохранять и отображать служебные поля для демонстрации;
- явно маркировать документ как проект.
