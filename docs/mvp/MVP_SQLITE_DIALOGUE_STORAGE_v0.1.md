# MVP SQLite dialogue storage v0.1

- Статус: концепция хранения динамической истории Demo MVP
- Дата: 2026-05-24
- Контур: Demo MVP

## Зачем нужен SQLite

В Demo MVP есть два разных типа контекста:

- статический контекст: роль, правила, workflow, статусы, ограничения;
- динамический контекст: история текущего диалога и structured result прошлых тактов.

Markdown-файлы подходят для статического контекста. SQLite подходит для простой динамической истории, потому что история меняется на каждом такте и не является документацией.

## Что хранит SQLite

На уровне смысла достаточно хранить:

- session/chat;
- message;
- turn output / structured result;
- last document draft.

Это не production storage design. В этом документе не фиксируются SQL-миграции, индексы, транзакционные гарантии, роли доступа или lifecycle документов.

## Session / chat

Смысл: отдельный пользовательский диалог.

Что важно для MVP:

- идентификатор сессии;
- время создания;
- краткий текущий статус;
- ссылка на последний structured result, если он есть.

## Message

Смысл: сообщение пользователя или ассистента.

Что важно для MVP:

- роль: user или assistant;
- текст сообщения;
- порядок в диалоге;
- время создания.

Для первых демо-сценариев достаточно вернуть в LLM короткую цепочку 5-10 последних сообщений или весь короткий диалог целиком.

## Turn output / structured result

Смысл: служебный результат ответа LLM за такт.

Что важно для MVP:

- workflow status;
- known facts;
- open questions;
- warnings;
- data statuses;
- document draft;
- structured JSON;
- next step.

Runtime может использовать эти данные для отображения, диагностики демо и будущего восстановления контекста.

## Last document draft

Смысл: последний сформированный проект ТК/ТТК в рамках сессии.

Что важно для MVP:

- хранить последний черновик отдельно от обычного текста сообщения;
- маркировать его как проект;
- не считать его юридически утвержденной картой;
- не проектировать полноценное версионирование документов в этом проходе.

## Что не хранить в SQLite в этом проходе

- Production state machine.
- Полноценный audit/event log.
- Сложный approval workflow.
- Интеграционные mapping под учетные системы.
- Секреты и API keys.
- Нормативную базу как runtime-данные.

## Что не хранить в markdown

- Историю user/assistant сообщений.
- Structured result каждого такта.
- Runtime state.
- Последний документ как изменяемое состояние диалога.

## Граница

SQLite в Demo MVP - локальный и простой способ сохранить диалоговую память прототипа. Это не обещание будущего production storage.
