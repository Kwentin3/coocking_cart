# MVP demo limitations v0.1

- Статус: ограничения Demo MVP
- Дата: 2026-05-24
- Контур: Demo MVP

## Главная граница

Demo MVP показывает идею продукта и проверяет ценность сценария. Он не является production-ready системой.

Результат ассистента - проект документа, который должен проверить технолог, руководитель или другое ответственное лицо предприятия.

## Ограничения результата

- Проект ТК/ТТК не является юридически финальной картой.
- MVP не проводит лабораторные исследования.
- MVP не подтверждает показатели безопасности.
- MVP не подтверждает сроки хранения без данных предприятия.
- MVP не подтверждает температурные режимы без данных предприятия.
- MVP не подтверждает нормы потерь без производственной проработки.
- MVP не заменяет производственный контроль и внутреннее утверждение.

## Ограничения диалога и контекста

- История user/assistant сообщений отправляется в LLM целиком только пока диалог короткий.
- Для первых демо-сценариев ориентир - 5-10 сообщений.
- Если диалог растет, нужен отдельный подход к сжатию или управлению памятью.
- Markdown context layers являются статическими и не хранят runtime state.
- LLM не должна автоматически создавать, переписывать или подключать новые context layers.

## Архитектурные ограничения

- Нет полноценной production state machine.
- Нет production semantic context layer.
- Нет managed state layer.
- Нет audit/event log production-уровня.
- Нет полноценного calculation engine.
- Нет validation layer production-уровня.
- Нет document versioning production-уровня.
- Нет approval workflow.

## Интеграционные ограничения

- Нет прямого импорта в iiko.
- Нет прямого импорта в r_keeper.
- Нет прямого импорта в StoreHouse.
- Нет прямого импорта в 1С.
- Structured JSON в MVP является нейтральным представлением, а не форматом конкретной учетной системы.

## Коммуникационные правила

Ассистент должен прямо показывать:

- что документ является проектом;
- какие данные подтверждены пользователем;
- какие данные рассчитаны;
- какие данные взяты из справочника;
- какие данные требуют проверки;
- какие данные требуют производственной проработки;
- какие поля не заполнены.

## Future track

После демонстрации и проверки продуктовой ценности можно отдельно проектировать production architecture: managed state, state machine, semantic context, calculation, validation, audit, versioning, approval и integrations.
