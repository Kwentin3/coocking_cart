# MVP markdown context layers v0.1

- Статус: описание markdown context layers Demo MVP
- Дата: 2026-05-24
- Контур: Demo MVP

## Назначение

Markdown context layers - это короткие рабочие слои, из которых runtime собирает статическую часть контекстного окна LLM.

Они не заменяют PRD, workflow, research или glossary. Каждый слой является сжатым производным материалом от common product baseline.

## Общие правила слоев

- Слой должен быть коротким и пригодным для вставки в LLM context.
- Слой должен описывать правила, а не историю обсуждений.
- Слой не должен хранить user/assistant историю.
- Слой не должен содержать production state machine.
- Слой не должен добавляться автоматически LLM.
- Изменение слоя должно быть осознанным документационным изменением.

## 00_AGENT_ROLE.md

Зачем нужен: задает роль ассистента и границы обещаний.

Что должно быть:

- кто такой ассистент;
- предметная область;
- какой результат он помогает получить;
- что он не должен обещать.

Чего не должно быть:

- длинного PRD;
- юридических трактовок;
- production-инструкций.

Производный источник: PRD v0.2, workflow v0.2, glossary.

## 01_PRODUCT_RULES.md

Зачем нужен: фиксирует поведенческие правила продукта.

Что должно быть:

- не выдумывать неподтвержденные данные;
- задавать уточняющие вопросы;
- показывать предупреждения;
- различать ТК и ТТК;
- вести к проекту документа.

Чего не должно быть:

- детальной нормативной базы;
- JSON-схемы;
- backend-логики.

Производный источник: PRD v0.2, research, decision log.

## 02_WORKFLOW.md

Зачем нужен: дает LLM дорожную карту диалога.

Что должно быть:

- идея блюда;
- уточнение формата;
- кулинарная гипотеза;
- корректировка;
- подтверждение состава;
- подтверждение технологии;
- выбор ТК/ТТК;
- формирование проекта;
- предупреждения;
- JSON.

Чего не должно быть:

- полный workflow-документ;
- UI-проектирование;
- state machine.

Производный источник: product workflow v0.2.

## 03_NORMATIVE_BASE_SHORT.md

Зачем нужен: дает краткий доменный контекст без длинного research.

Что должно быть:

- ГОСТ 31987-2012 как ориентир структуры технологических документов;
- различие ТК и ТТК;
- санитарный контур СанПиН;
- безопасность пищевой продукции в контуре ТР ТС 021/2011;
- финальное утверждение за предприятием.

Чего не должно быть:

- длинных цитат;
- спорных правовых трактовок;
- обещаний юридической проверки.

Производный источник: research по нормативной базе v0.1 и addendum v0.2.

## 04_TECH_CARD_STRUCTURE.md

Зачем нужен: задает ожидаемые блоки проекта ТК/ТТК.

Что должно быть:

- наименование;
- область применения;
- сырье;
- рецептура;
- брутто/нетто;
- выход;
- технология;
- подача;
- хранение и реализация;
- пищевая ценность;
- предупреждения.

Чего не должно быть:

- production data model;
- SQL-схем;
- интеграционного mapping.

Производный источник: PRD v0.2, research, glossary.

## 05_DATA_STATUSES.md

Зачем нужен: заставляет ассистента явно показывать степень доверия к данным.

Что должно быть:

- подтверждено пользователем;
- рассчитано системой;
- взято из справочника;
- требует проверки;
- требует производственной проработки;
- не заполнено.

Чего не должно быть:

- сложного lifecycle документов;
- approval workflow;
- production state transitions.

Производный источник: PRD v0.2, workflow v0.2, decision log.

## 06_OUTPUT_RULES.md

Зачем нужен: задает правила ответа пользователю и structured output.

Что должно быть:

- когда задавать вопросы;
- когда формировать проект;
- что включать в итоговую выдачу;
- как показывать warnings и statuses;
- как относиться к structured JSON.

Чего не должно быть:

- глубокой production JSON-схемы;
- frontend-макетов;
- интеграционных адаптеров.

Производный источник: PRD v0.2, workflow v0.2, structured output contract.

## 07_DEMO_LIMITATIONS.md

Зачем нужен: удерживает LLM внутри демо-границ.

Что должно быть:

- demo mode;
- не production-ready;
- не юридическое утверждение;
- не лабораторное подтверждение;
- нет прямого импорта;
- короткая история диалога;
- нет полноценного state machine.

Чего не должно быть:

- production roadmap в деталях;
- обещаний будущих функций как текущих возможностей.

Производный источник: PRD v0.2, audit, recommendation, decision log.

## context_manifest.yml

Зачем нужен: задает порядок сборки markdown context pack.

Что должно быть:

- список markdown layers;
- короткие описания;
- указание, что порядок важен;
- указание, что диалоговая история добавляется runtime отдельно.

Чего не должно быть:

- динамических сообщений пользователя;
- секретов;
- backend config;
- environment-specific settings.

Производный источник: MVP file-driven context architecture и context window contract.
