# Decision log

- Дата старта: 2026-05-20
- Статус: активный журнал продуктовых решений

## Принятые решения

| ID | Дата | Решение | Обоснование | Последствия |
| --- | --- | --- | --- | --- |
| D-001 | 2026-05-20 | В первом MVP продукт формирует проект документа, а не юридически финально утвержденную карту. | Безопасность, соответствие требованиям и фактические производственные данные должны подтверждаться предприятием. | В интерфейсе и документах нужен явный статус "проект" и блок предупреждений. |
| D-002 | 2026-05-20 | Для пользователя используется простая формулировка "технологическая карта", но система должна различать ТК и ТТК по характеру блюда. | Пользователь часто мыслит общим термином, а продукт должен помогать выбрать корректный тип документа. | В workflow нужен шаг определения: стандартное блюдо, новое блюдо или адаптация. |
| D-003 | 2026-05-20 | Для авторских и адаптированных блюд по умолчанию предполагается ТТК. | ТТК лучше подходит для новой и нетрадиционной продукции, где важны требования к сырью, безопасности, хранению и пищевой ценности. | В PRD и workflow авторские блюда ведутся к проекту ТТК, если пользователь не подтвердил обратное. |
| D-004 | 2026-05-20 | Первый MVP не делает прямой импорт в iiko, r_keeper, 1С или StoreHouse, а формирует нейтральный структурированный JSON для будущих интеграций. | Прямые интеграции требуют маппинга номенклатуры, единиц измерения, прав и правил учета конкретной системы. | Интеграционные адаптеры остаются следующим этапом после проверки продуктовой ценности. |
| D-005 | 2026-05-20 | Фокус PRD - продуктовая ценность и workflow, без инженерной архитектуры. | Документ должен быть понятен владельцу кафе, технологу, повару, инвестору или партнеру. | PRD не описывает внутренние механизмы реализации, инфраструктуру и архитектурные детали. |
| D-006 | 2026-05-20 | PRD v0.2 остается продуктовым документом без описания инженерной архитектуры. | Документ нужен для обсуждения ценности, сценария и границ ответственности с нетехническими участниками. | В v0.2 усилены workflow, аудитория, ограничения и статусы данных, но не добавлены детали реализации. |
| D-007 | 2026-05-20 | MVP учитывает доступность, редкость и дороговизну ингредиентов, но не считает полноценную себестоимость. | Пользователь может хотеть более доступные ингредиенты, но закупочные цены, поставщики и складской учет требуют отдельного контура. | В PRD себестоимость вынесена из MVP, а замены ингредиентов оставлены как продуктовая помощь. |
| D-008 | 2026-05-20 | В документе должны отображаться статусы данных. | Пользователь должен понимать, какие значения подтверждены, рассчитаны, взяты из справочника или требуют проверки. | В PRD и workflow v0.2 добавлен блок статусов данных. |
| D-009 | 2026-05-20 | Производственная проработка выносится в vNext. | Реальные потери, выход, температура, время и упаковка проверяются фактическим приготовлением, а не только диалогом. | В PRD v0.2 добавлено направление развития после MVP. |
| D-010 | 2026-05-20 | Продукт формирует проект документа, а не лабораторно или юридически подтвержденную карту. | Лабораторные исследования, сроки годности, показатели безопасности и утверждение находятся в зоне ответственности предприятия. | В PRD v0.2 усилена граница ответственности и список неподтверждаемых системой данных. |
| D-011 | 2026-05-20 | В PRD нужен короткий executive summary для первичного обсуждения продукта. | Полный PRD полезен для детализации, но партнерам и знакомым сначала нужен быстрый обзор идеи, ценности, MVP и ограничений. | В начало PRD добавлен раздел "Коротко для обсуждения". |
| D-012 | 2026-05-20 | Термин "canonical JSON" не используется в основном продуктовом PRD как обязательный термин. | Для нетехнических читателей понятнее формулировка "структурированный JSON для будущих интеграций". | В PRD и workflow продуктовый текст смягчен; техническое уточнение можно оставить для будущего blueprint. |
| D-013 | 2026-05-20 | После MVP стоит рассмотреть отдельный режим технолога. | Технологу нужен не только финальный текст, а список спорных полей, фильтры по статусам, возможность подтверждать значения и возвращать проект на доработку. | В PRD и workflow добавлено направление "режим технолога" после MVP. |
| D-014 | 2026-05-20 | Выходной результат должен показываться как набор рабочих блоков, а не как один сплошной текст. | Пользователь должен быстро увидеть документ, таблицу ингредиентов, предупреждения, статусы данных и JSON. | В PRD и workflow добавлен пример выходного результата со статусами данных. |
| D-015 | 2026-05-24 | В репозитории будет два документационных контура: Demo MVP и future production architecture. | Быстрый демонстрационный прототип и будущая production-система имеют разные цели, риски и глубину проектирования. | Документация должна разделять `docs/mvp/` и `docs/production/`, используя общие product docs как baseline. |
| D-016 | 2026-05-24 | Для Demo MVP принимается file-driven context architecture. | Для демонстрации важнее быстро управлять контекстом и правилами без внедрения полноценного semantic context engine. | MVP-документы должны описать сборку context pack из файлов и отделить этот подход от production architecture. |
| D-017 | 2026-05-24 | Статические части контекстного окна управляются markdown-файлами, а не хардкодом в коде. | Роль ассистента, правила продукта, workflow, статусы данных и demo limitations должны редактироваться как документационные артефакты. | Нужно создать `docs/mvp/context/` с управляемыми markdown context layers и manifest. |
| D-018 | 2026-05-24 | Динамическая история диалога хранится в SQLite, а не в markdown-файлах. | История user/assistant сообщений является runtime state, а не документацией или статическим prompt asset. | MVP должен разделять static markdown context и dynamic dialogue history. |
| D-019 | 2026-05-24 | Каждый такт MVP отправляет в LLM markdown context pack и короткую цепочку user/assistant сообщений. | Для демонстрации достаточно отправлять 5-10 последних сообщений целиком, не строя сложную memory/state architecture. | Контракт контекстного окна MVP должен явно описать состав prompt context и ограничение длины истории. |
| D-020 | 2026-05-24 | Structured output используется уже в MVP, но без production-сложности. | Даже демо должно возвращать машинно-разбираемый результат, warnings и статусы данных, но не обязано иметь полную production-схему. | Нужен отдельный `MVP_STRUCTURED_OUTPUT_CONTRACT_v0.1.md` с минимальным контрактом ответа. |
| D-021 | 2026-05-24 | Production state machine и semantic context layer остаются future track. | Их детальная проработка сейчас замедлит демонстрационный MVP и смешает быстрый прототип с production-проектированием. | Production-документы создаются отдельно и не являются блокером для MVP context architecture. |
| D-022 | 2026-05-24 | LLM не должна автоматически создавать и подключать новые context layers в Demo MVP. | Авторасширение слоев размоет контроль над prompt/context assets и может незаметно изменить поведение демо. | Изменение `docs/mvp/context/` выполняется как осознанное документационное изменение, а не runtime-действие модели. |

## Как обновлять журнал

Новые решения добавляются отдельной строкой с датой, коротким обоснованием и последствиями. Если решение отменено или заменено, его не нужно удалять: добавляется новая запись со ссылкой на предыдущее решение.
